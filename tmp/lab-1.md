# ЭТАП 1: Генерация и отбор идей
## 1. Фиксация ограничений проекта
**Ограничения:**
- Срок: 10 недель
- Команда: до 8 человек
- Обязательное использование LLM
- Результат: MVP (один ключевой сценарий)
- Ограниченный бюджет (LLM API + хостинг)

**Целевая аудитория:**
- Студенты IT-направлений
- Junior-разработчики
- Люди, проходящие технические интервью

**Цель проекта:**
Создать MVP AI-продукта с реальной ценностью и демонстрацией навыков разработки интеллектуальной системы.

## 2. Генерация ≥10 идей
**Использованы методы:**
- Мозговой штурм  
- SCAMPER  
- Частично «обратный штурм»  

#### Идеи (12 штук)
**1. AI Interview Simulator (технический)**  
Симуляция технического интервью с анализом ответов.  
**2. AI Code Reviewer**   
LLM анализирует код как senior-разработчик и объясняет архитектурные ошибки.  
**3. AI Resume Analyzer**  
Проверка резюме разработчика, оценка шансов пройти screening.  
**4. AI “Объясни мой код”**  
Пользователь вставляет код — система объясняет, что он делает, и находит слабые места.  
**5. AI Algorithm Coach**  
Пошаговая помощь в решении алгоритмических задач (не даёт ответ сразу, а направляет).  
**6. AI Stack Skill Analyzer**  
Тестирует знания по конкретному стеку (React, Django, Node).  
**7. AI System Design Trainer**  
Тренажёр по системному дизайну (с вопросами и разбором).  
**8. AI Debugging Simulator**  
Даёт багнутый код — пользователь ищет проблему, LLM анализирует ход мыслей.  
**9. AI Peer Programming Bot**  
Имитация работы в паре с разработчиком.  
**10. AI Interview Anxiety Trainer**  
Симуляция стрессового интервью (таймер, уточняющие вопросы).  
**11. AI Technical Quiz Generator для вузов**  
Генерация контрольных/тестов по программированию.  
**12. AI Progress Tracker для подготовки к интервью**  
Система, которая анализирует историю задач и строит карту слабых тем.

# Этап 2: Проработка концепции
## 1. Формулировка проблемы (в проверяемом виде)

**Кто пользователь:**
Студенты 2–4 курса, джуны и мидлы, готовящиеся к техническому собеседованию (JS / Python).

**Ситуация:**
Им нужно подготовиться к live-интервью, но:
- нет практики формата «вопрос → устный/письменный ответ → уточняющие вопросы»
- нет персональной обратной связи
- сложно понять, какие темы западают  

Что сейчас неудобно:
- LeetCode даёт задачи, но не симулирует интервью
- YouTube даёт теорию, но нет интерактива
- Друзья не всегда могут провести mock-интервью
- Настоящее интервью — стресс, нет тренировки формата

## 2. Актуальность vs востребованность
**Актуальность (почему сейчас):**
- Рост конкуренции на junior позиции
- Массовое использование LLM в обучении
- Удалённые собеседования стали стандартом
- Популярность платформ вроде LeetCode и HackerRank

**Востребованность (кому нужно)**
- Студенты IT-направлений
- Самоучки
- Junior-разработчики
- Люди, меняющие стек (например, с Python на JS)

## 3. Сегменты аудитории
**Студенты**:
- Нужна базовая подготовка
- Боль: не понимают уровень ожиданий

**Джуны**:
- Готовятся к первому офферу
- Боль: страх провала

**Мидлы**:
- Готовятся к переходу в более сильную компанию
- Боль: нужно подтянуть алгоритмы

**4. Альтернативные решения**
| Альтернатива              |         Недостатки                       |
|---------------------------|------------------------------------------|
|   LeetCode                | Нет имитации живого интервью             |
|   YouTube	                | Нет интерактивности                      |
| Реальный mock с человеком	| Дорого или неудобно                      |
| ChatGPT вручную	        | Нет структурированного анализа прогресса |

# Этап 3: Валидация и MVP
## 1. Ключевые гипотезы
- Пользователи хотят симуляцию «как на реальном интервью».
- Им важнее разбор слабых мест, чем просто правильность ответа.
- Они готовы использовать такой сервис регулярно (2–3 раза в неделю).

## 2. Моделирование интервью (пример)**
Вопрос респонденту: _Как ты сейчас готовишься к интервью?_
Типичный ответ: _«Решаю задачи, но не понимаю, как меня будут спрашивать вживую.»_

Боль:
- Нет тренировки формулирования ответа
- Нет оценки soft-аспекта объяснения

## 3. MVP — один ключевой сценарий
Сценарий:
- Пользователь выбирает стек (например, JavaScript)
- Система генерирует вопрос по алгоритмам
- Пользователь пишет ответ

LLM:
- анализирует решение
- оценивает корректность
- оценивает объяснение
- даёт разбор
- выделяет слабые темы
- сохраняется результат

Без видеосвязи. Без real-time. Только текстовая симуляция.

Это полностью реализуемо за 10 недель.

# Этап 4: Технологии и архитектура
## 1. Технологический стек
Frontend:
- React / Next.js
- TypeScript
- Tailwind
- Telegram Web App

Почему:
- Быстрая разработка
- Хорошо для портфолио
- Простая интеграция с API

Backend:
- Node.js + Express / NestJS
или
- Python + FastAPI

База данных:
- PostgreSQL
- Prisma / SQLAlchemy

## 2. Роль LLM
LLM используется для:
- Генерации вопросов
- Анализа кода
- Анализа объяснения
- Формирования фидбэка
- Выявления слабых тем

Пример промпта:
```Python
"""
Ты технический интервьюер уровня Senior.
Оцени ответ кандидата по 5 критериям:
- корректность
- оптимальность
- понимание сложности
- качество объяснения
- пробелы в знаниях
"""
```

## 3. Архитектура

Frontend (React)
        ↓
Backend API (FastAPI)
        ↓
LLM API
        ↓
PostgreSQL

Дополнительно:
- Модуль статистики
- Хранение истории интервью
- Простая система скоринга
